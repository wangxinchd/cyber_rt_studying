## 1. 概述

`cyber/croutine` 是 Apollo Cyber RT 的**协程（Coroutine）调度子系统**。核心目标是**在单线程内实现高效的并发任务切换**，避免线程创建和上下文切换的高昂开销，同时让开发者可以用"同步顺序代码"风格编写异步逻辑。

### 1.1 协程 vs 线程

| 维度 | 线程（Thread） | 协程（Coroutine） |
|---|---|---|
| 调度方式 | OS 抢占式（进内核） | 协作式（用户态，主动让出） |
| 上下文切换开销 | ~1000-10000 cycles，进内核 | ~200-500 cycles，纯用户态 |
| 资源占用 | 每个线程独立内核对象 | 每个协程仅一个 2MB 栈 |
| 数量上限 | 几十~几百 | 可达数千甚至数万 |
| 通信方式 | 锁、信号量等 | 消息队列、Channel 等 |

### 1.2 协程的核心优势

- **无进入内核**：上下文切换只涉及寄存器保存/恢复
- **海量并发**：单线程可运行成千上万个协程
- **同步写法，异步性能**：代码读起来像顺序执行，实际上是事件驱动的异步执行
- **内存局部性好**：同一线程内的协程共享 CPU 缓存

### 1.3 协程的调用过程与栈管理

理解协程的关键在于理解**"栈"在哪里、切换时栈发生了什么"**。

当一个线程运行协程时，它的执行状态 = **寄存器 + 栈**。其中寄存器保存当前指令位置、临时变量等，而栈保存函数调用链（局部变量、返回地址等）。

**协程切换时发生了什么？** 当协程主动调用 `Yield()` 让出 CPU 时，`ctx_swap` 汇编函数将当前寄存器快照保存到 `RoutineContext`（包含一个 2MB 独立栈），然后恢复下一个协程的寄存器快照和栈指针 `sp`。协程 A 的栈和协程 B 的栈完全独立，互不干扰。

**为什么每个协程需要独立栈？** 调度器本身是一个 `for(;;)` 循环跑在某个 OS 线程上。如果协程复用调度器的栈空间，协程 A 的局部变量会覆盖协程 B 的局部变量——而且调度器的栈容量有限（~8MB），协程稍微深递归就会溢出。因此每个协程必须持有自己的栈空间，`RoutineContext` 就是这块 2MB 独立内存区域的容器。

```text
调度器线程（OS 管理栈 ~8MB）
└── 调度循环 for(;;) { NextRoutine(); Resume(); }
    ↑
    │ ctx_swap 切换回来
    │
协程 A（独立 2MB 栈）
├── Process() 局部变量
├── DeepFunc() 调用链
└── ...
```

## 2. 核心数据结构

### 2.1 RoutineContext — 协程执行上下文

每个协程有自己独立的执行上下文，包含独立栈空间和寄存器快照区。

```cpp
// code/cyber/croutine/detail/routine_context.h

constexpr size_t STACK_SIZE = 2 * 1024 * 1024;  // 2MB 独立栈
#if defined __aarch64__
constexpr size_t REGISTERS_SIZE = 160;            // ARM: 27个寄存器 × 8字节
#else
constexpr size_t REGISTERS_SIZE = 56;             // x86_64: 7个寄存器 × 8字节
#endif

struct RoutineContext {
  char stack[STACK_SIZE];   // 独立栈空间
  char* sp = nullptr;       // 栈指针，切换时保存/恢复
};
```

- 每个协程拥有 **2MB 独立栈**，完全隔离
- 栈顶附近预置了返回地址和参数，用于首次切入协程
- `sp`（栈指针）指向当前栈顶，是切换的核心

#### 为什么每个协程必须有独立大栈？

不同协程实现选择不同的栈管理策略，各有权衡：

| 设计 | 做法 | 优点 | 缺点 |
|---|---|---|---|
| Cyber RT（独立大栈） | 每个协程一个 2MB 栈 | 实现简单，栈完全隔离，协程内可递归、深调用 | 内存占用大（1万个协程 ≈ 20GB） |
| 共享栈（Go goroutine） | 多个协程共享一个小栈区 | 内存利用率高 | 切换时需要复制栈数据，边界处理复杂 |
| 分段栈（早期 Go） | 协程从小区间开始，递归时扩展 | 内存按需分配 | 需要编译器插入栈检查，无法纯 C++ 实现 |

Cyber RT 是 **C++ 嵌入式/车载框架**，优先"实现简单 + 确定性"，而不是极致内存效率。2MB 栈对车载感知算法完全合理，协程数量通常几十到几百个，总内存占用可控。如果需要支持数十万协程（服务器场景），应该选择 Go 风格的共享栈方案。

### 2.2 CRoutine — 协程对象

```cpp
// code/cyber/croutine/croutine.h

enum class RoutineState {
    READY,      // 可执行，等调度器选中
    FINISHED,   // 执行完毕
    SLEEP,      // 主动睡眠（带超时唤醒时间 wake_time_）
    IO_WAIT,    // 等待 I/O
    DATA_WAIT   // 等待数据（消息）
};

class CRoutine {
public:
    // 协程入口函数类型
    using RoutineFunc = std::function<void()>;
    using Duration = std::chrono::microseconds;

    // 静态接口（操作当前协程）
    static void Yield();
    static void Yield(const RoutineState &state);
    static void SetMainContext(const std::shared_ptr<RoutineContext> &context);
    static CRoutine *GetCurrentRoutine();
    static char **GetMainStack();

    // 协程操作
    bool Acquire();                  // 获取锁（防止多调度器同时运行）
    void Release();                  // 释放锁
    RoutineState Resume();           // 切入协程执行
    RoutineState UpdateState();      // 更新状态（唤醒超时的 SLEEP 等）
    void Sleep(const Duration &d);   // 睡眠一段时间
    void HangUp();                   // 挂起等待数据

    // ... getter/setter ...

private:
    std::string name_;
    std::chrono::steady_clock::time_point wake_time_;  // 睡眠唤醒时间
    RoutineFunc func_;                                   // 协程体函数
    RoutineState state_;                                // 当前状态
    std::shared_ptr<RoutineContext> context_;           // 执行上下文（独立栈）

    std::atomic_flag lock_;                             // 获取锁
    std::atomic_flag updated_;                          // 异步事件更新标记

    // 线程局部变量 —— 每个线程独立
    static thread_local CRoutine *current_routine_;    // 当前运行的协程
    static thread_local char *main_stack_;             // 主线程栈
};
```

## 3. 上下文切换原理

这是整个协程系统的核心。Cyber 使用**纯手写汇编**实现上下文切换，而非 POSIX ucontext 接口，原因有三：

1. **ucontext 不可移植** — Linux 已废弃，macOS 彻底移除
2. **粒度更细** — 只保存协程真正用到的寄存器（caller-saved），不浪费
3. **性能最优** — 零抽象，每条指令都是必需的

### 3.1 x86_64 汇编实现

```asm
# code/cyber/croutine/detail/swap_x86_64.S

.globl ctx_swap
.type  ctx_swap, @function
ctx_swap:
      pushq %rdi              # ① 保存 caller-saved 寄存器到当前栈
      pushq %r12
      pushq %r13
      pushq %r14
      pushq %r15
      pushq %rbx
      pushq %rbp
      movq %rsp, (%rdi)       # ② 把当前栈指针保存到 (*src_sp)

      movq (%rsi), %rsp       # ③ 切换到目标栈：rsp = (*dest_sp)
      popq %rbp               # ④ 从目标栈弹出之前保存的寄存器
      popq %rbx
      popq %r15
      popq %r14
      popq %r13
      popq %r12
      popq %rdi
      ret                     # ⑤ 返回 → 跳到目标协程上次中断的位置继续
```

**参数约定**（System V AMD64 ABI）：
- `%rdi` → `src_sp`（当前栈指针要存到这里）
- `%rsi` → `dest_sp`（要切换到的目标栈指针）

**切换过程图解**：

```
切换前（协程A在运行）                  切换后（调度器主栈在运行）

┌─────────────────┐                 ┌─────────────────┐
│   协程A 栈(2MB)  │                 │   协程A 栈(2MB)  │
│                 │                 │   ···           │
│   [rdi] [r12]   │                 │   [rdi] [r12]   │
│   [r13] [r14]   │                 │   [r13] [r14]   │
│   [r15] [rbx]   │    ctx_swap     │   [r15] [rbx]   │
│   [rbp] ◄ rsp   │ ──────────────► │   [rbp]         │
│   ret addr     │                 │   ret addr      │
└─────────────────┘                 └─────────────────┘

┌─────────────────┐                 ┌─────────────────┐
│   主栈(调度器)   │                 │   主栈(调度器)   │
│                 │                 │                 │
│   [rdi] [r12]   │                 │   [rdi] [r12]   │
│   [r13] [r14]   │                 │   [r13] [r14]   │
│   [r15] [rbx]   │                 │   [r15] [rbx]   │
│   [rbp]         │                 │   [rbp] ◄ rsp   │
│   ctx_swap ret  │                 │   ctx_swap ret  │
└─────────────────┘                 └─────────────────┘
```

### 3.2 C++ 包装

```cpp
// code/cyber/croutine/detail/routine_context.h

inline void SwapContext(char** src_sp, char** dest_sp) {
    ctx_swap(reinterpret_cast<void**>(src_sp),
             reinterpret_cast<void**>(dest_sp));
}
```

### 3.3 协程栈初始化（首次切入）

```cpp
// code/cyber/croutine/detail/routine_context.cc

// 栈布局：
//              +------------------+
//              |    Reserved       |
//              +------------------+
//              |  Return Address  |  ← 入口函数 CRoutineEntry
//              +------------------+
//              |       RDI        |  ← 参数：CRoutine* this
//              +------------------+
//              |       R12        |
//              +------------------+
// ctx->sp =>   |       ...        |
//              +------------------+
//              |       RBP        |
//              +------------------+
void MakeContext(const func &f1, const void *arg, RoutineContext *ctx) {
    // 1. 初始化寄存器快照区为0
    ctx->sp = ctx->stack + STACK_SIZE - 2 * sizeof(void*) - REGISTERS_SIZE;
    std::memset(ctx->sp, 0, REGISTERS_SIZE);

    // 2. 在栈顶预置返回地址（函数指针）
    char *sp = ctx->stack + STACK_SIZE - 2 * sizeof(void*);
    *reinterpret_cast<void**>(sp) = reinterpret_cast<void*>(f1);  // CRoutineEntry
    sp -= sizeof(void*);

    // 3. 预置参数（CRoutine* this）
    *reinterpret_cast<void**>(sp) = const_cast<void*>(arg);
}
```

关键点：在栈上"伪造"一个函数调用帧，使得 `ctx_swap` 切换后 `ret` 指令会跳到 `CRoutineEntry(this)` 入口函数。

## 4. 协程生命周期

### 4.1 创建

```cpp
// code/cyber/croutine/croutine.cc

CRoutine::CRoutine(const std::function<void()> &func) : func_(func) {
    // 从对象池获取 RoutineContext（避免频繁 new/delete）
    context_ = context_pool->GetObject();
    if (context_ == nullptr) {
        // 超过池上限，警告后直接 new
        context_.reset(new RoutineContext());
    }

    // 用 MakeContext 初始化栈，入口函数是 CRoutineEntry
    MakeContext(CRoutineEntry, this, context_.get());
    state_ = RoutineState::READY;
}

// 统一入口函数
void CRoutineEntry(void *arg) {
    CRoutine *r = static_cast<CRoutine*>(arg);
    r->Run();                                // 执行用户传入的 func_
    CRoutine::Yield(RoutineState::FINISHED); // 结束后主动让出
}
```

### 4.2 切入（Resume）

```cpp
// code/cyber/croutine/croutine.cc

RoutineState CRoutine::Resume() {
    if (force_stop_) {
        state_ = RoutineState::FINISHED;
        return state_;
    }
    if (state_ != RoutineState::READY) {
        AERROR << "Invalid Routine State!";
        return state_;
    }

    current_routine_ = this;                           // 设置当前协程
    SwapContext(GetMainStack(), GetStack());           // 切走：主栈 ← 协程栈
    current_routine_ = nullptr;                         // 切回后清除
    return state_;
}
```

### 4.3 让出（Yield）

```cpp
// code/cyber/croutine/croutine.h

inline void CRoutine::Yield(const RoutineState &state) {
    auto routine = GetCurrentRoutine();
    routine->set_state(state);  // 设置新状态
    // 切回主栈，控制权回到调度器
    SwapContext(GetCurrentRoutine()->GetStack(), GetMainStack());
}

inline void CRoutine::Yield() {
    // 无状态变更的让出
    SwapContext(GetCurrentRoutine()->GetStack(), GetMainStack());
}
```

### 4.4 睡眠与挂起

```cpp
inline void CRoutine::Sleep(const Duration &sleep_duration) {
    wake_time_ = std::chrono::steady_clock::now() + sleep_duration;
    CRoutine::Yield(RoutineState::SLEEP);
}

inline void CRoutine::HangUp() {
    CRoutine::Yield(RoutineState::DATA_WAIT);
}
```

## 5. 调度器与协程的协作

调度器（位于 `cyber/scheduler` 模块）驱动协程的执行循环：

```
┌──────────────────────────────────────────────────────────────┐
│                    调度器线程（主栈）                          │
│                                                              │
│  for (;;) {                                                  │
│      for (CRoutine* r : ready_queue) {                      │
│          r->Acquire();       // 加锁，防止被多调度器同时运行   │
│          r->Resume();        // 切入协程                      │
│          // ← 此时在协程栈执行用户代码，直到 Yield() 才返回    │
│                                                              │
│          if (r->state() == READY)   put back to ready queue │
│          else if (r->state() == SLEEP)    put to sleep queue │
│          else if (r->state() == DATA_WAIT) put to wait queue│
│          else if (r->state() == FINISHED)   recycle          │
│          r->Release();         // 解锁                         │
│      }                                                        │
│  }                                                            │
└──────────────────────────────────────────────────────────────┘

┌──────────┐    ctx_swap    ┌──────────────────┐
│ 主栈     │ ◄─────────────► │ 协程 A 栈(2MB)  │
│ (调度器) │                 │ Run() → 用户代码  │
└──────────┘                 └──────────────────┘
      ▲                           Yield()
      │
─────┼───────────────────────────────────────────────
      │
┌────┴──────┐  ctx_swap   ┌──────────────────┐
│ 主栈      │ ◄──────────► │ 协程 B 栈(2MB)   │
└───────────┘              └──────────────────┘
      ▲
      └─────────── 不断循环选择下一个 READY 协程切入
```

调度器负责：
1. 维护多个协程队列（READY、SLEEP、DATA_WAIT 等）
2. 每轮循环调用 `UpdateState()` 唤醒到期的 SLEEP 和 DATA_WAIT 协程
3. 依次对 READY 队列中的协程执行 `Acquire() → Resume() → Release()`
4. 根据协程 `Yield` 后的状态决定放到哪个队列

## 6. 状态更新机制

```cpp
// code/cyber/croutine/croutine.h

inline RoutineState CRoutine::UpdateState() {
    // 同步事件：检查 sleep 超时
    if (state_ == RoutineState::SLEEP &&
        std::chrono::steady_clock::now() > wake_time_) {
        state_ = RoutineState::READY;
        return state_;
    }

    // 异步事件：检查数据/IO 事件是否就绪
    // atomic_flag updated_ 用于防止重复唤醒
    if (!updated_.test_and_set(std::memory_order_release)) {
        if (state_ == RoutineState::DATA_WAIT ||
            state_ == RoutineState::IO_WAIT) {
            state_ = RoutineState::READY;
        }
    }
    return state_;
}
```

- `updated_.test_and_set()` 是**一次性触发器**：唤醒后自动设置，调度器感知到状态变化后需要调用 `SetUpdateFlag()` 重置
- 区分**同步**（SLEEP 带超时）和**异步**（DATA_WAIT/IO_WAIT 靠事件触发）两种唤醒路径

## 7. RoutineFactory：协程工厂

`CreateRoutineFactory` 模板函数为带 `DataVisitor` 的组件生成"消息驱动的协程函数"，将消息消费逻辑封装为协作式循环。

### 7.1 单消息模板

```cpp
// code/cyber/croutine/routine_factory.h

template <typename M0, typename F>
RoutineFactory CreateRoutineFactory(
    F&& f,
    const std::shared_ptr<data::DataVisitor<M0>>& dv) {
    RoutineFactory factory;
    factory.SetDataVisitor(dv);
    factory.create_routine = [=]() {
        return [=]() {
            std::shared_ptr<M0> msg;
            for (;;) {                            // 无限循环
                CRoutine::GetCurrentRoutine()->set_state(RoutineState::DATA_WAIT);
                if (dv->TryFetch(msg)) {          // 尝试获取消息
                    f(msg);                        // 消费消息
                    CRoutine::Yield(RoutineState::READY);
                } else {
                    CRoutine::Yield();             // 无消息，让出控制权
                }
            }
        };
    };
    return factory;
}
```

### 7.2 执行流程

```
组件回调函数
    │
    ▼
CRoutine::Run() → 用户写的 Process(msg) { ... }
    │
    ├── 有消息 → Process(msg) → Yield(READY) → 被调度器重新放入 ready_queue
    │
    └── 无消息 → Yield() → 切回调度器 → 等待 DATA_WAIT 被唤醒
```

这就是 Cyber 组件编写者写的回调函数能"看起来同步等待消息"的原因——内部就是一个无限循环 + `TryFetch` + `Yield`。

### 7.3 多消息模板

工厂支持 1~4 个消息模板参数（`CreateRoutineFactory<M0, F>` 到 `CreateRoutineFactory<M0, M1, M2, M3, F>`），分别对应多通道数据融合场景。逻辑相同，只是 `TryFetch` 变成多参数版本。

## 8. x86_64 vs AArch64 汇编实现对比

Cyber 为不同 CPU 架构提供了独立的汇编实现，原理相同但保存的寄存器不同：

### 8.1 x86_64（Intel/AMD 桌面/服务器）

```asm
# code/cyber/croutine/detail/swap_x86_64.S

ctx_swap:
      pushq %rdi
      pushq %r12
      pushq %r13
      pushq %r14
      pushq %r15
      pushq %rbx
      pushq %rbp
      movq %rsp, (%rdi)        # 保存当前栈指针

      movq (%rsi), %rsp        # 切换到目标栈
      popq %rbp
      popq %rbx
      popq %r15
      popq %r14
      popq %r13
      popq %r12
      popq %rdi
      ret
```

### 8.2 AArch64（ARM 嵌入式/车载）

```asm
# code/cyber/croutine/detail/swap_aarch64.S

ctx_swap:
    stp    x0,   x30, [sp,#-16]!   # 保存 x0 和链接寄存器
    stp    d8,   d9, [sp,#-16]!
    stp    d10,  d11, [sp,#-16]!
    stp    d12,  d13, [sp,#-16]!
    stp    d14,  d15, [sp,#-16]!    # 浮点寄存器
    stp    x1,   x19, [sp,#-16]!
    stp    x20,  x21, [sp,#-16]!
    stp    x22,  x23, [sp,#-16]!
    stp    x24,  x25, [sp,#-16]!
    stp    x26,  x27, [sp,#-16]!
    stp    x28,  x29, [sp,#-16]!

    mov    x3,   sp
    str    x3,   [x0]             # 保存当前栈指针

    ldr    x3,   [x1]             # 加载目标栈指针
    mov    sp,   x3

    ldp    x28,  x29,  [sp]       # 恢复寄存器
    ldp    x26,  x27,  [sp,#16]!
    ldp    x24,  x25,  [sp,#16]!
    ldp    x22,  x23,  [sp,#16]!
    ldp    x20,  x21,  [sp,#16]!
    ldp    x1,   x19,  [sp,#16]!
    ldp    d14,  d15,  [sp,#16]!
    ldp    d12,  d13,  [sp,#16]!
    ldp    d10,  d11,  [sp,#16]!
    ldp    d8,   d9,   [sp,#16]!
    ldp    x0,   x30,  [sp,#16]!

    add    sp,   sp,   #16        # 平衡栈
    ret
```

### 8.3 对比总结

| 维度 | x86_64 | AArch64 |
|---|---|---|
| 文件 | `swap_x86_64.S` | `swap_aarch64.S` |
| 保存的通用寄存器 | rdi, r12-r15, rbx, rbp（7个） | x0-x30（31个） |
| 保存的浮点寄存器 | 无 | d8-d15（8个，64位） |
| 总寄存器大小 | 56 bytes | 160 bytes |
| 使用指令 | `push/pop/mov` | `stp/ldp`（成对操作） |
| 适用场景 | 桌面 x86_64 / AMD64 | 车载 ARM（如 NVIDIA Jetson） |

### 8.4 为什么 AArch64 保存更多寄存器？

ARM AAPCS 调用约定中，调用者保存（caller-saved）寄存器远多于 x86_64。x86_64 只要求保存 `rbx, rbp, r12-r15, rdi`（caller-saved 中的 callee-saved 部分），而 ARM 的调用约定更复杂，因此 AArch64 版本保存了更多上下文。

#### Choreography 调度策略详解

**Choreography** 直译为"舞蹈编排"，是 Cyber RT 的第二种调度策略。它的核心思想是：**每个协程在创建时就绑定一个固定的 Processor（OS 线程）**，明确知道自己该在哪个核上运行，而不是所有 Processor 共享一个全局队列。

**Classic（经典）** 类似餐厅公共取餐区，所有厨师共享同一批菜篮子，谁有空谁去拿——协程不知道自己最终落在哪个 Processor 上。

**Choreography（编排）** 类似给每个厨师分配固定工位，A 厨师只用自己的 1 号食材篮，B 厨师只用 2 号——协程通过 `processor_id` 固定在某个核上。

| 维度 | Classic（经典） | Choreography（编排） |
|---|---|---|
| 协程分配方式 | 放到共享队列，谁有空谁抢 | 协程绑定固定 `processor_id`，放入对应 Processor 专属队列 |
| 运行队列 | 所有 Processor 共享一个 `cr_group_` | 每个 Processor 有独立的 `cr_queue_` |
| 调度确定性 | 低（取决于竞争） | 高（协程固定在某个核上） |
| 典型场景 | 一般组件、读者、通用任务 | 实时任务、对 CPU 核有亲和性需求的场景 |

Choreography 特别适合车载实时系统——例如感知模块的协程固定跑在 CPU 核 0，定位模块固定跑在核 1，确保关键任务不会被其他任务抢走处理器。

```cpp

```cpp
// code/cyber/scheduler/policy/scheduler_choreography.cc

// DispatchTask: 根据是否指定了 processor_id 选择不同路径
void SchedulerChoreography::DispatchTask(const std::shared_ptr<CRoutine>& cr) {
  if (cr->processor_id() >= 0) {
    // 有固定 processor 亲和性 → 放入 ChoreographyContext
    ChoreographyContext::Enqueue(cr, cr->processor_id());
  } else {
    // 无亲和性 → 放入 ClassicContext 池
    ClassicContext::Enqueue(cr);
  }
}

// NotifyProcessor: 同样通过 SetUpdateFlag 唤醒

  auto cr = FindCRoutineInMap(crid);
  if (cr->state() == RoutineState::DATA_WAIT ||
      cr->state() == RoutineState::IO_WAIT) {
    cr->SetUpdateFlag();
    // 根据 processor_id 选择唤醒路径
    if (cr->processor_id() >= 0) {
      ChoreographyContext::Notify(cr->processor_id());
    } else {
      ClassicContext::Notify(cr->group_name());
    }
  }
}

#### Processor — 处理器线程（实际执行协程的地方）

```cpp
// code/cyber/scheduler/processor.cc

// Processor::Run() — 每个 Processor 线程的无限循环
void Processor::Run() {
  while (!stop_) {
    // 从上下文获取下一个可运行的协程
    std::shared_ptr<CRoutine> croutine = context_->NextRoutine();
    if (croutine == nullptr) {
      context_->Wait();   // 无可运行协程，等待唤醒
      continue;
    }

    // 切入协程执行（ctx_swap 切换栈）
    croutine->Resume();

    // 此时控制权已切回（协程调用了 Yield）
    croutine->Release();
  }
}
```

`Processor` 是真正执行 `Resume()` 的地方。每个 `Processor` 对应一个 OS 线程，它不断从运行队列取协程、`Resume()` 切入执行、`Yield()` 让出后继续下一轮。

#### ClassicContext — 经典调度上下文

```cpp
// code/cyber/scheduler/policy/classic_context.cc

// 全局运行队列：group_name → (priority → 协程队列)
static std::unordered_map<std::string,
    std::array<std::deque<std::shared_ptr<CRoutine>>, 16>> cr_group_;

// NextRoutine: 遍历 group/priority，返回第一个 READY 协程
std::shared_ptr<CRoutine> ClassicContext::NextRoutine() {
  for (auto& group_pair : cr_group_) {
    for (int prio = 15; prio >= 0; --prio) {
      auto& queue = group_pair.second[prio];
      for (auto it = queue.begin(); it != queue.end(); ++it) {
        if ((*it)->Acquire()) {              // 原子获取锁
          (*it)->UpdateState();               // 检查是否可运行
          if ((*it)->state() == RoutineState::READY) {
            return *it;                        // 返回给 Processor
          }
          (*it)->Release();                    // 仍不可运行，释放
        }
      }
    }
  }
  return nullptr;
}

// Notify: 唤醒某个 group 的 Processor
void ClassicContext::Notify(const std::string& group_name) {
  auto& cv = cv_wq_[group_name];
  cv.notify_one();
}

#### ChoreographyContext — 编排调度上下文

```cpp
// code/cyber/scheduler/policy/choreography_context.cc

// 每个 processor 一个优先队列
std::vector<std::multimap<uint32_t,
    std::shared_ptr<CRoutine>, std::greater<uint32_t>>> cr_queue_;

// NextRoutine: 优先取高优先级协程
std::shared_ptr<CRoutine> ChoreographyContext::NextRoutine() {
  auto& queue = cr_queue_[processor_id_];
  for (auto it = queue.begin(); it != queue.end(); ++it) {
    if (it->second->Acquire()) {
      it->second->UpdateState();
      if (it->second->state() == RoutineState::READY) {
        return it->second;
      }
      it->second->Release();
    }
  }
  return nullptr;
}

#### Task API — 公开的任务接口

```cpp

// 协程感知的挂起
inline void Yield() {
  if (auto r = CRoutine::GetCurrentRoutine()) {
    r->Yield();  // 协程内：协作式让出
  } else {
    std::this_thread::yield();  // 非协程：直接让出线程

}

#### TaskManager — 任务管理线程池

// code/cyber/task/task_manager.cc

// 构造时创建 worker 协程池
TaskManager::TaskManager() {
  for (int i = 0; i < num_threads_; ++i) {
    // 创建协程工厂，每个 worker 协程从队列取任务执行
    auto factory = CreateRoutineFactory([this]() {
      for (;;) {
        std::function<void()> task;
        if (task_queue_.Pop(100ms, task)) {
          task();              // 执行任务
        } else {
          // 队列空，HangUp 让出（等待新任务）
          CRoutine::GetCurrentRoutine()->HangUp();
        }
      }
    });
    scheduler::Instance()->CreateTask(factory, "task_pool_worker");
  }
}

void TaskManager::Enqueue(const std::function<void()>& task) {

### 9.6 I/O 模块（`cyber/io`）

#### PollHandler — 异步 I/O 与协程的桥梁

```cpp
// code/cyber/io/poll_handler.cc

// Block(): 在协程内等待 I/O 就绪
void PollHandler::Block(int fd, bool write) {
  auto routine = CRoutine::GetCurrentRoutine();
  routine_ = routine;

  // 注册到 epoll
  poller_->Add(fd, write ? EPOLLOUT : EPOLLIN,
      [this](const PollResponse& resp) {
        // I/O 就绪后，唤醒协程
        scheduler::Instance()->NotifyTask(routine_->id());
      });

  // 协程挂起，等待 I/O
  routine_->Yield(RoutineState::IO_WAIT);
}

// ResponseCallback(): epoll 回调中唤醒协程
void PollHandler::ResponseCallback(const PollResponse& resp) {
  if (routine_ && routine_->state() == RoutineState::IO_WAIT) {
    routine_->Wake();
    scheduler::Instance()->NotifyTask(routine_->id());
  }
}
```

`PollHandler` 让协程可以非阻塞地等待文件描述符就绪——协程 `Yield(IO_WAIT)` 挂起，epoll 回调唤醒后重新 `Resume()`。

### 10.1 Component 创建到执行的完整链路

```
用户代码
  │
  ▼
Component::Initialize(config)
  │
  ├── ① 创建 DataVisitor<M0>（关联 ChannelBuffer + Notifier）
  │
  ├── ② CreateRoutineFactory<M0>(Proc回调, dv)
  │     │
  │     └── 工厂内部生成函数:
  │         for (;;) {
  │             set_state(DATA_WAIT)
  │             if (TryFetch(msg)) { Proc(msg); Yield(READY); }
  │             else                 { Yield(); }
  │         }
  │
  └── ③ scheduler->CreateTask(factory, nodeName)
        │
        ├── ④ std::make_shared<CRoutine>(func)
        │     ├── 分配 2MB 栈
        │     ├── MakeContext(CRoutineEntry, this, ctx)
        │     └── state_ = READY
        │
        ├── ⑤ id_cr_[id] = croutine
        │
        └── ⑥ DispatchTask(cr) → ClassicContext::Enqueue()
              │
              └── ⑦ ClassicContext::Notify(group_name)
                    │
                    └── ⑧ Processor::Run() 循环中被唤醒

--- 运行时 ---

数据到达 ChannelBuffer
  │
  ├── ⑨ ChannelBuffer::HasData() → dv->Notify()
  │     └── notifier_() → scheduler->NotifyProcessor(cr_id)
  │
  ├── ⑩ Scheduler::NotifyProcessor()
  │     ├── cr->SetUpdateFlag()   // 重置 updated_ 原子锁
  │     └── ClassicContext::Notify(group_name)  // 唤醒 Processor 线程
  │
  └── ⑪ Processor::Run()
        │
        ├── NextRoutine() → cr
        ├── cr->Acquire()
        ├── cr->UpdateState() → state_ = READY   // 检测到 updated_
        └── cr->Resume()

              ├── ctx_swap(main_stack ← cr_stack)
              │
              └── 进入协程栈执行：

                  │
                  ├── TryFetch(msg) 成功
                  ├── Proc(msg)     执行用户逻辑
                  └── Yield(READY)  让出
                        │
                        ├── ctx_swap(cr_stack ← main_stack)
                        └── 返回 Processor::Run()，等待下次 Resume
```

### 10.2 协程状态迁移表

| 当前状态 | 触发事件 | 下一状态 | 所在模块 |
|---|---|---|---|
| — | `std::make_shared<CRoutine>` | `READY` | scheduler |
| `READY` | `Processor::Resume()` | 执行中 | processor |
| 执行中 | `Yield()` | `READY` | croutine |
| 执行中 | `Yield(DATA_WAIT)` | `DATA_WAIT` | croutine |
| 执行中 | `Yield(SLEEP)` | `SLEEP` | croutine |
| 执行中 | `Yield(IO_WAIT)` | `IO_WAIT` | croutine |
| 执行中 | `func_()` 返回 | `FINISHED` | croutine |
| `DATA_WAIT` | `SetUpdateFlag()` + `Notify()` | `READY` | scheduler |
| `IO_WAIT` | `Wake()` + `Notify()` | `READY` | scheduler/poll_handler |
| `SLEEP` | `wake_time_` 到期 | `READY` | processor |
| 任意 | `cr->Stop()` | `FINISHED` | scheduler |

### 10.3 调度策略对比

| 维度 | SchedulerClassic（经典） | SchedulerChoreography（编排） |
|---|---|---|
| 核心理念 | 共享队列，竞争式调度 | 固定亲和性，确定性调度 |
| 运行队列 | `map<group, array<16, deque<CRoutine>>>` | `vector<multimap<prio, CRoutine>>` |
| 队列顺序 | 每 group 下 16 级优先级（0-15） | 每 processor 独立 multimap，高优先级在前 |
| 分组方式 | 按 `group_name`（可配） | 按 `processor_id`（可选亲和性） |
| 协程分配 | 放入共享队列，谁有空谁取 | 绑定固定 `processor_id`，放入对应队列 |
| 唤醒路径 | `ClassicContext::Notify(group_name)` | `ChoreographyContext::Notify(processor_id)` |
| 典型用途 | 一般组件、读者、通用任务 | 对 CPU 核有亲和性需求的实时任务 |
| 典型场景举例 | 感知、定位等通用模块 | 感知固定核 0，定位固定核 1 |
